Modif RS set_bases du 06-10-2013 Version 2.16.001
--------------------------------------------------------------------------------------------------------- 
- AMELIORATION DE LA RESTAURATION SQL : 
       - La restauration des très gros fichiers SQL se fait beaucoup mieux et avec une meilleure
         gestion des indications de progression. La durée d'intégration de gros fichiers SQL
         peut prendre plusieurs heures. Il est donc utile de prévoir le temps que cela va mettre.

       - A des fins d'évaluation si echec d'une restauration :
        ° Le dernier INSERT INTO est écrit sur le disque dans le fichier 'LastInsert.sql'.
        ° Le plus grand INSERT INTO est écrit sur le disque dans le fichier 'MaxLastInsert.sql'.

       - A des fins de réglages et pour s'adapter aux grands blocs de données il est possible de
         définir la taille du buffer qui sera utilisé pour lire les données sql par l'entrée
         suivante dans le fichier 'set_bases.ini'
         [Parametres]
             sql_buffer = 15000000

Modif RS set_bases du 17-06-2013 Version 2.16.000
--------------------------------------------------------------------------------------------------------- 
- BUG :  - De l'export et import Datasemp
         Ce bug provenait de la façon dont set_bases faisait un dump des bases Datasemp.
         En effet le dump classique sans option générait des INSERT INTO de plus de 500 Mo.
         Cela faisait parfois exploser les capacités d'une QString pour stocker la requète
         INSERT INTO lue, coupait les INSERT INTO n'importe où.
         le tout variable selon les machines (mémoire dispo, config SQL etc...),
         
         Pour faire le Dump mysql une option net_buffer_length= permet de limiter
         la taille des INSERT INTO à des tailles plus raisonnables, en générant
         plusieurs INSERT INTO de tailles plus réduites au lieu d'un gigantesque
         INSERT INTO ingérable. Ce bug fourbe faisait que sur certaines machines
         qgetdatasemp pouvait intégrer le Vidal Datasemp et sur d'autres non.  

         Le problème se règle avec encore deux nouvelles options :
         situées dans deux fichiers à la racine du binaire de set_bases.
         (Attention : ne pas mettre d'espaces avant ou après le signe =)
         Les valeurs indiquées semblent être un bon compromis entre la taille
         maximum des données rencontrées (longs textes) pouvant être inclue dans
         un INSERT INTO. Si trop petit le dump coupera l'INSERT INTO si trop grand
         l'INSERT INTO ne pourra pas être incorporé lors de la restauration.

       - option d'export de Dump_IndexMedica --quote-name -O net_buffer_length=4096
       à placer dans le fichier /set_bases_bin/Dump_IndexMedica_options.opt
       - option d'export de Dump_Datasemp --quote-name -O net_buffer_length=128000
       à placer dans le fichier /set_bases_bin/Dump_Datasemp_options.opt
